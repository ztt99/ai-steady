一、什么是“分层职责”？

分层职责本质是：

每一层只做一类事情，不越界，不混杂。

它解决的问题是：

降低耦合

提高可维护性

提高可测试性

方便扩展

如果不分层，最后就会变成：

一个 2000 行的 index.ts

那是灾难。

二、你这个项目的分层是怎么设计的？

我们按“工程架构层次”来讲。

① CLI 层（接口层）

文件：

src/cli.ts

职责：

读取命令行参数

调用核心逻辑

输出结果到终端

它不应该：

解析 AST

扫描文件

计算统计数据

它只是入口适配层。

② 应用协调层（Application Layer）

文件：

src/index.ts

职责：

串联 scanner + analyzer

组织整体流程

汇总报告

它不应该：

递归读文件

写 AST 逻辑

打印 CLI

它只是 orchestration（调度层）。

可以理解为：

总导演

③ 基础设施层（I/O 层）

文件：

scanner/fileScanner.ts

职责：

读取文件系统

递归扫描目录

返回文件路径数组

它只做一件事：

把“磁盘上的东西”变成“程序可处理的数据”

它不应该：

分析 AST

统计函数数量

输出日志

④ 领域逻辑层（Domain Layer）

文件：

analyzer/fileAnalyzer.ts

职责：

创建 SourceFile

遍历 AST

统计变量/函数/类

返回结构化结果

它是：

项目的核心业务逻辑

它不应该：

读取磁盘

打印 console

处理 CLI 参数

⑤ 类型定义层（模型层）

文件：

types/report.ts

职责：

定义 Report 结构

定义 Analyzer 输出格式

例如：

export interface FileReport {
  filePath: string;
  functionCount: number;
  variableCount: number;
}

它不包含逻辑。

三、你这个分层属于什么架构？

这其实是一个非常典型的：

分层架构（Layered Architecture）

逻辑依赖方向应该是：

CLI
 ↓
Application
 ↓
Domain
 ↓
Infrastructure

不能反过来。

例如：

❌ analyzer 里不能去读命令行
❌ scanner 里不能去统计函数

五、依赖方向必须单向

这是专业工程的核心规则：

cli
  ↓
core
  ↓
scanner / analyzer


只要满足下面三条，就没有架构问题：

cli.ts 只做输入输出

index.ts 只做 orchest
scanner 和 analyzer 不互相依赖


规则和框架的边界，
如果未来要支持：某些文件忽略这个规则，是规则自己判断注释？还是 analyzer 统一处理？

- 是规则自己判断注释
  - 规则耦合了语法层和策略层
  - 需要每个规则都判断
- 还是 analyzer 统一处理？


- rule 只负责发现问题
- engine 统一过滤是否忽略
